Git 学习笔记

工作区（working directory）→ 持有实际文件
版本库（repository）→ 隐藏目录 .git → 包含称为 stage 或 index 的暂存区, 还有 Git 为我们自动创建的第一个分支 main, 以及指向 main 的第一个指针 HEAD，所以 git add命令实际上是把要提交的所有修改放到暂存区 stage，然后执行 git commit 就可以一次性把暂存区的所有修改提交到分支。

分支

除非将分支推送到远程仓库，否则该分支就是不为他人所见的。

当 Git 无法自动合并分支时，就必须首先解决冲突，解决冲突后才能再提交，合并。解决冲突就是把 Git 合并失败的文件手动编辑为我们希望的内容。

通常合并分支时，Git 会用 Fast forward 模式合并分支，在这种模式下，合并后删除分支，会丢失分支信息。

如果强制禁用 Fast forward 模式，Git 就会在合并时生成一个新的 commit，这样就可以从分支历史上看到分支信息。

分支策略：main 分支非常稳定，仅用来发布新版本，平时不能在上面干活，当新版本确定发布时，再把 dev 分支合并到 main 分支上；干活都在 dev 分支上，每个人都有自己的分支。

Bug分支

修复 bug 时，通过创建新的 bug 分支进行修复，然后合并，最后删除该分支。

当手头工作没有完成时，先把工作现场通过 git stash 进行保存，然后去修复 bug，修复后，再 git stash pop，回到工作现场。

Feature分支

开发一个新 feature ，最好新建一个分支。如果要丢弃一个没有被合并过的分支，通过 git branch -D 分支名 强行删除。

多人协作

首先，尝试用 git push 远程库名 分支名 推送自己的修改。如果推送失败，可能是因为远程分支比本地的版本更新，需要先用 git pull 尝试进行合并；如果合并有冲突，则需要先解决冲突，并在本地提交；如果没有冲突或者解决掉冲突后，再用 git push 远程库名 分支名 推送就能成功。

标签管理

发布一个版本时，通常先在版本库中打一个标签（tag)，这样就可以唯一确定打标签时刻的版本，Git 的标签其实是指向某个 commit 的指针，tag 即为一个让人容易记住的有意义的名字（eg: v1.2)， 它跟某个 commit 绑在一起。

标签总是和某个 commit 挂钩，如果这个 commit 既出现在 main 分支，又出现在 dev 分支，那么在这两个分支上都可以看到这个标签。

配置Git

忽略特殊文件

某些文件必须要放到 Git 工作目录中，但又不能提交它们，例如保存了数据库密码的配置文件等，每次 git status 都会显示 Untracked files··· 要解决这个问题，可以在 Git 工作区的根目录下创建一个特殊的 .gitignore 文件，然后把要忽略的文件名填进去，Git 就会自动忽略这些文件。

原则：

①忽略操作系统自动生成的文件，例如缩略图等；

②忽略编译生成的中间文件、可执行文件等，即如果一个文件是通过另一个文件自动生成的，那么自动生成的文件就没必要放进版本库，比如 Java 编译生成的 .class 文件。

③忽略带有敏感信息的配置文件，例如存放口令的配置文件。

.gitignore 文件本身要放到版本库里，并且可以对.gitignore 做版本管理。

配置别名

每个仓库的 Git 配置文件都放在.git/config文件中.要删除别名，直接在配置文件中把相应的行删掉即可.

当前用户的 Git 配置文件放在用户主目录下的一个隐藏文件 .gitconfig 中，配置别名也可以直接修改这个文件。
